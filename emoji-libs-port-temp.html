Emoji-Libs

My Challenge

Emoji-Libs was my final project as part of my front-end web development class. It's part of my portfolio because it was unique opportunity to create content, design website, and actually bring it to life with code. Of all the projects in my portfolio I think this did the most to increase my understanding of the big picture, all the roles within tech, and how design can best work within them. This portfolio piece reviews both elements of the websites design as well as lessons learned as part of creating content and coding the site.

My Role

This project was 100% my work product, including concept creation, content, design, and code. Throughout the process our TAs were available to help and 5+ users also helped with feedback for concept and design.

Brainstorm Ideas

The Emoji Bible
Learn how to use emojis in 3 different ways: emoji-libs, express your feelings with emojis, and things you have always wanted to say in emojis

Idiom Translator
Translates English Idioms into Spanish using Google translate API. Also has animated cat that slides in after each idiom translation & says "Ole"

Travel Pics & Music
Shows you travel pics from anywhere in the world with accompanying local music. Uses Instagram tags and Spotify and Instagram APIs

Discovery

They say code is akin to language. I speak 2.25 languages andhave taught language. I would say that analogy holds up, to a point. Having no prior coding experience, with this class what was for sure was that I was learning a new paradigm - a new way to way to think, work, and communicate with my computer.

HTML: a skeleton made of tupperware

When I first began learning about HTML I would get this picture in my mind of Tupperware. I saw the "divs" as the containers of the content (photos, text, etc.), the space around the containers is the margin, and the space around the food/content inside these containers is the padding. I also like the metaphor of HTML as a skeleton that you attach content too, so essentially HTML is a skeleton made of Tupperware.

CSS is hard

My front-end development teacher, Avand, had this bad habit of saying: "that's easy." And like that annoying Staples "easy button" him saying it didn't actually make it so. For my final project I was so focused on making it work I left making it look good until the end. Then Avand said: "you can't save CSS to the end. CSS is hard!" This for me was a huge victory as I had finally gotten Avand to admit that something was hard. But, all this to say, that nothing in code can ever be taken for granted as "easy." Sometimes even the most basic things can trip you up, causing you to loose huge chunks of time.

CSS is much more than a pretty face. CSS does so much more than just add color and styling. Many people prefer even CSS over JavaScript for animations.

In CSS, he who speaks last, speaks loudest. This can be a blessing and a curse. A blessing in that you can easily overwrite things by just changing the order. The curse is that sometimes you have multiple classes and styles (especially on main elements: table, etc.) that can screw you up without you knowing why.

CSS may never give you why or where. With some languages, like JavaScript will tell you exactly where the error is and maybe even a bit about why its not working. But with CSS you are mostly on your own. Good luck.

Time, time, time is not on your side, no it's not. Some of the "easiest" things in CSS can take hours to fix, even for seasoned developers.

JavaScript is harder

And what's even worse, the analogies. Somehow visualizing variables as a "bucket" and functions as a "sausage factory" - is supposed to help you to understand inherently difficult concepts. It doesn't.

Everything shuts down. Unlike CSS/HTML when JavaScript is not working every bit of your code just stops working, even parts that have nothing to do with the error. Good times. Good times.

Gives you a clue about the where if not also the why (sort of). JavaScript tells you the exact place where the error starts, the actual line. It also gives you a bit of information about what caused the error although it's usually in computer speak like: something is "null," "undefined is not a function," or "unexpected token ILLEGAL" - am I going to coding jail?

JavaScript is a very powerful tool. It was not lost on me that almost anything you want to do in front-end web development can be done in JavaScript.

Knowing some code is understanding all code

My new paradigm. I recently attended a Ruby pairing workshop. I had exactly zero experience in Ruby or any other back-end code for that matter. I asked my pair what was the difference between Ruby and Ruby on Rails. I will paraphrase his answer in the style of the SAT: Ruby is to Rails as JavaScript is to jQuery. I had no experience with Ruby, but understood him completely. I know I have just cracked the surface, but any code learning going forward should be much easier for me given my new paradigm.

Computers are stupid, but never wrong

I can almost hear the collective grown of designers when they get to this part. "Oh no! No actual Code!" I cut this bit of code out of my Emoji-Libs site to demontstrate how I understand that computers think. Designed by mathematicians, computers "think" in a way that is not very obvious to most of us. There is a common phrase among coders "computer are stupid, but never wrong." This is to say that you have to tell them what you want in explicit detail in a lanaguage that only they understand to get the output you want.

Line 1: function chooseNouns(nounEvent){

This is the start of the function that allows you to choose a noun emoji from the list of options in the noun table. The noun events is the input passed in the function (aka sausage factory)

Line 2: var imgClicked = nounEvent.target;

This a variable that remembers the emoji (shoe, etc.) you selected in the noun table so it can later be placed within the story text.

Line 4: var tableClicked = $(imgClicked).parents('table');

Remembering that you are in the noun table so later you can clear the noun table.

Line 6-14: var clonedImg= document.createElement("img");

Create the new image that will be placed in the text of the story with all of the attributes it needs: image, classes, the translation text (i.e. pig emoji translates to the word "piggy").

Line 16: clickedNounEl.textContent = "";

Clears out the old content you had in that space, in this case the word "noun" that was underlined in the text.

Line 18: clickedNounEl.appendChild(clonedImg);

You think that may be it right? What else is there to do? No. You actually have to tell the computer to append your new emoji image to the page. You could get all the way here and think you were done, but you would never see it because you did not tell it to append to the page.

Line 20: clearTable(tableClicked);

Oh, and one more thing. Now you have to tell the computer to remove that table of nouns now that you don't need it anymore.

Phew. I am exhausted. And that was just one tiny piece of the code that makes up Emoji-Libs

Synthesis

Responsive web

- Bootstrap is helpful with nav, footer, and scroll tables but it requires a lot of work on visuals.

- Bootstrap's grid can help with layout, but you often need to insert media queries to make things work.

- Using percentages and global margins will help to make responsive easier.

Break things

- People often do not want to change their code for fear of breaking it, especially when JavaScript brings your whole page to a screeching halt.

- But the only way to learn is to change things, break things, and then learn to fix them. (Making commits on GitHub so you can “undo” is a great practice too).

Code is Creative

- Many people think that code is just about being a “monkey,” rote implementation of someone else’s creativity.

- For this project I was in a unique position: wrote content, designed, and coded. And I found that code was very creative and even inspired better design.

MVP is good

- Focus on completing a MVP first, get it done.

- You may find that the lean UX implied by MVP may also result in better, crisper design.

Make it work before you Make it pretty

- This phrase should probably be tattooed on the arm of any web developer or designer.

- While also true for design, in code it's especially true as the penalty for making it “pretty” too soon can be devastating, many hours of wasted time that might completely be lost.

Always do the process

- This project was for a front-end web development class. I didn’t need to look at the strategic implications of this project, sketch designs, create digital wireframes, user test - any of that.

- But for me, doing the design process is necessary for any kind of decision making.

Timing is tough

- How long will that take? I can just see asking this questions as a designer/portfolio manager. And now I know the answer may not be that easy. I have learned a lot about what is hard and what is easy to do in code, but even seasoned developers are hit with unexpected issues that can drastically effect time to completion

Some that happened to me are:

+ The “easy CSS” problem that took 3 hours to fix.

+ The problem I didn’t know I had. Hey why is your website look like X, Y, Z on my Android…

+ I am just going to finish one more thing… is it 4am?

Scope creep

- Scope creep is one of the toughest things to manage in code.

- I just need to do one more thing…

- Wouldn't it be cool if…

- STOP! Define the scope and stick to it. Don’t let a small project become the Taj Mahal.

- What helps me is to fall back on the concept of MVP. What is the minimum I need to do to get this out? The rest goes on your Trello to do list. And it will be a long list.

The problem & the Emoji-Libs solution

Problem

People often get confused with the meaning of Emojis and misuse them leading to minor mix ups and even relationship breakups.

Solution

Emoji-Libs gives you real examples so you can practice how to use Emojis properly improving your Emoji Intelligence Quotient (or EIQ) dramatically overtime.

Understand that anything can happen, aka that mystifying 3 hours you lost to centering.

2

Pitch in

3

Knowledge is power

- Sometimes I feel like dealing with engineers is like dealing with a mechanic, "don't worry about what's under the hood little missy."

- When designers know code they can take back some of that power.

3

Code can inspire/influence design

- No matter how perfect the design nothing renders quite the same on Sketch/Illustrator as it does on the web page, especially considering responsive web and the hundreds of different sized devices

- Sometimes a coding accident or a need to modify the content to fit the code will lead to a better, cleaner design

Useability Testing

User feedback improved UI design

Users saw the initial design as "clunky" as it went back and fourth between text and cell phone texting bubbles. Due to their suggestions, we introduced the "monkey" narrator and that fixed the flow.

User's prerogative to change their mind

User's mentioned that they might want to change their choice of emojis after they made a selection. I modified my JavaScript so they could make more than one choice.

Users loved the "history of the emoji"

User's love the "History of the Emoji" section that appeared in the original draft of the Emoji Bible. Orginally it was cut, but found a way to repackage it, making it the "About" page in Emoji-Libs

Build & Deliver

In addtion to some of the general lessons learned about code and design I also learned some "tricks" that made things easier. Keep in mind that tricks are different than "hacks". Hacks, like using negative margin or XXX, can be very dangerous when it comes to responsive web and usually way more trouble then they are worth.

Tricks of the Trade

Reflections, aka the Zen of Code

1

The best design comes from multi-disciplinary teams

There seems to be so many stories of developers and designer at odds with each other, but this project made it clear why we all need to work togther, as only then can we inspire each other to make the best product.

2

Waste not, want not

Don't worry if good content or design elements get cut. Figure out a smart way to repackage them and use them again. And when it comes to code tweaking and reusing is the key to faster page building.

3

There is nothing like seeing your creation come to life

Even the most beautiful design is still just flat. It's not dynamic. It doesn't having any functionality. When you build a site, with you own two hands and see it functioning it's an amazing feeling. I have been a writer, artist, and film maker and belive that, like those mediums, code is truly creative.
