Emoji-Libs

My Challenge

Emoji-Libs was my final project as part of my front-end web development class. I chose to include it as part of my portfolio because it is a unique opportunity for most designers to create content, design website, and actually bring it to life with code. Of all the projects in my portfolio I think this did the most to increase my understanding of the big picture of all the roles within tech and how design can best work within them.

My Role

This project was 100% my work product, including concept creation, content, design, and code. Throughout the process our TAs were available to help when I got stuck and 3-5 users also helped with feedback for concept and design.

Research & Discovery

They said code is akin to language. I speak 2.25 languages and have taught language and I would say that analogy holds up, to a point. Having no prior coding experience, with this class what was for sure was that I was learning a new paradigm - a new way to way to think, work, and communicate with my computer.

HTML a skeleton made of tupperware

When I first began learning about HTML I would get this picture in my mind of Tupperware. I saw the "divs" as the containers of the content (photos, text, etc.), the space around the containers is the margin, and the space around the food/content inside these containers is the padding. I also like the metaphor of HTML as a skeleton that you attach content too, so essentially HTML is a skeleton made of Tupperware.

CSS is hard

My front-end development teacher Avand had this bad habit of saying "it's easy." And like that annoying Staples "easy button" him saying it was easy didn't actually make it so. For my final project I was so focused on making it work I left making it look good until the end. Avand said: "you can't save CSS to the end. CSS is hard!" This for me was a huge victory as I had finally gotten Avand to admit that something was hard. But, all this to say, that nothing in code can ever be taken for granted as "easy." Sometimes even the most basic things can trip you up, causing you to loose huge chunks of time.

Key items to note about css:

CSS is much more than a pretty face. CSS now does so much more than just add color and styling. Many people prefer even CSS over java script for animations.

In CSS, he who speaks last, speaks loudest. This can be a blessing and a curse. A blessing in that you can easily overwrite things by just changing the order. The curse is that sometimes you have multiple classes and styles (especially on main elements: table, etc.) that can screw you up without you knowing why.

CSS may never give you why or where. With some languages, like Java Script will tell you exactly where the error is and maybe even a bit about why its not working. But with CSS you are mostly on your own. Good luck.

Time, time, time is not on your side, no its not. Some of the "easiest" things in CSS can take hours to fix, even for seasoned developers.

Java Script is harder

And what's even worse, the analogies. Somehow visualizing variables as a "bucket" and a function as a "sausage factory" - is supposed to help you to understand inherently difficult concepts. It doesn't.

Key items to note about Java Script:

Everything shuts down. Unlike CSS/HTML when Java Script is not working every bit of your code just stops working, even parts that have nothing to do with the error. Good times. Good times.

Gives you a clue about the where if not also the why (sort of). Java Script tells you the exact place where the error starts, the actual line. It also gives you a bit of information about what caused the error although it’s usually in computer speak like: something is "null," "undefined is not a function," or "unexpected token ILLEGAL" - am I going to coding jail?

Java Script is a very powerful tool. It was not lost on me that almost anything you want to do in front-end web development can be done in Java Script.

Knowing some code is understanding all code

My new paradigm. I recently attended a Ruby pairing workshop. I had exactly zero experience in Ruby. I asked my pair what was the difference between Ruby and Ruby on Rails. I will paraphrase his answer in the style of the SAT: Ruby is to Rails as Java Script is to jQuery. I had no experience with Ruby, but understood him completely. I know I have just cracked the surface, but any code learning going forward should be much easier for me given my new paradigm.

User Research

User feedback improved UI design

Users saw the initial design as "clunky" as it went back and fourth between text and cell phone texting bubbles. Due to their suggestions, we introduced the "monkey" narrator and that fixed the flow.

User's prerogative to change their mind

User's mentioned that they might want to change their choice of emojis after they made a selection. I modified my Java Script so they could make more than one choice.

Users loved the "history of the emoji"

User's love the "History of the Emoji" section that appeared in the original draft of the Emoji Bible. Orginally it was cut, but found a way to repackage it, making it the "About" page in Emoji-Libs
